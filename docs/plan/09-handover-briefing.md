# 09 — 인수인계 브리핑: 설계 철학, 변화 지도, 이후 발전 수칙

> 계획 세션이 사용자에게 남기는 문서. 이후 설계를 발전·보완·수정할 때(어떤 모델과 일하든) 이 문서의 §4~6이 변경 관리의 기준이다.

## 1. 이 설계가 하려는 일 (한눈에)

당신은 이미 "AI 회사"의 **사옥**(대시보드 3장)과 **출퇴근 시스템**(iMac 디스패처, 1분 폴링+정기 회차)을 지었다. 강의 24장이 가르치는 것은 그 회사가 **안에서 실제로 일 잘하는 법**이고, 대조 결과 빠져 있던 것은 다섯 가지다:

1. **회사의 서고(기억)** — LLM Wiki. 지금은 읽은 논문의 보고서가 낱개로 쌓일 뿐, "읽을수록 똑똑해지는" 축적 구조가 없다.
2. **업무 매뉴얼의 품질 기준** — 스킬 12종은 이미 실전급이지만, 입력·출력·판단 기준·실패 처리가 고르게 적혔는지 점검한 적이 없고 원본이 버전관리되지 않는다.
3. **직원별 직무기술서** — 역할(목회자·부장·리뷰어…)은 있는데 "무엇을 해야 하고 무엇을 하면 안 되는지"(책임·금지사항)가 명문화되어 있지 않다.
4. **안전 수칙** — "커밋·금전·삭제는 보류"라는 좋은 관례가 UI 문자열에만 있다.
5. **돌수록 좋아지는 순환(Loop)** — 정기 회차는 "반복"이지 "개선"이 아니다. 초안→리뷰→수정→재검토처럼 품질이 누적되는 회로가 없다.

계획(02)의 P1~P6는 이 다섯 개를 순서대로 채우고, 마지막에 "PDF 한 편 넣으면 다음 연구 액션 제안까지" 종단 시나리오로 증명한다.

**핵심 설계 결정 3가지와 이유**:
- **repo가 단일 진실원(SSOT)**: 여러 디바이스·여러 세션이 같은 규칙을 봐야 하므로, 규약·스킬 원본·명세는 버전관리되는 한 곳(GitHub)에 두고 iMac은 "실행 장소"로 격하시킨다. 새 맥 추가 = clone 한 번.
- **빌더와 검수자의 분리**: 실무(Sonnet 5)와 감사(Opus 4.8)를 다른 세션으로 나누고, 각자 자기 장부(07/08)에만 쓰게 한다. 한 세션이 일하고 같은 세션이 스스로 합격 판정하는 구조를 금지 — 이것이 모델 능력보다 결과 신뢰성을 좌우한다.
- **원장(07) 중심의 불사(不死) 설계**: 세션은 죽는다는 전제. 상태는 대화가 아니라 파일에 산다. 어떤 세션이 죽어도 같은 킥오프를 새 세션에 붙이면 원장의 마지막 지점부터 이어진다.

## 2. 당신의 시스템이 어떻게 바뀌는가 (전 → 후)

**연구(구약학)**: [전] PDF → 분할 → NotebookLM 업로드 → 보고서(여기서 끝). [후] 보고서가 Wiki 페이지(요약·비판적 읽기 템플릿)로 축적되고, passages(성경 본문별) 축으로도 검색되며, 축적분에서 주기적으로 **연구 질문·공백 리포트**가 나오고, 논문 초안은 Writing Reviewer가 논리·근거를 점검한다. 문헌이 늘수록 시스템이 더 유용해지는 복리 구조.

**사역(희성교회 청소년부)**: [전] 행사 오케스트레이션은 강력하지만, 사역팀 연구 보드의 요청은 "연구 스킬 대기"로 사람을 기다린다. [후] 보드 요청이 자동으로 Literature Reviewer를 거쳐 보드에 회신되고(ministry.research 완성), **설교 준비가 글쓰기 Loop**(본문→초안→리뷰→수정→재검토)를 탄다 — 논문과 같은 부품을 쓰므로 유지보수는 한 곳.

**사업/개발(Hubrary·VoiceReading)**: [전] 구현·검증 버튼이 따로 논다. [후] 실험 Loop(구현→검증→결과 기록→다음 제안)로 명문화되고 worklog가 표준 스키마를 갖는다.

**인프라(가장 큰 변화)**: [전] iMac이 꺼지면 전체 정지하고 아무도 모름, 엔진 구조는 문서 제로(버스 팩터 1), 스킬 원본 위치는 사용자도 모름. [후] ENGINE.md(엔진 설계도 — 새 맥에서 재구축 가능), 하트비트(원격에서도 생사 확인), catch-up(밀린 회차 자동 소화), 스킬 전체 repo 버전관리, 경로 표준화·디바이스 프로파일.

## 3. 이후 커스텀 빌드업 — "무엇이든 추가하는" 표준 레시피

새 업무를 시스템에 넣고 싶을 때(예: 채점 시즌 자동화 강화, 새 강의 준비, 교회 새 부서, Hubrary 마케팅 리서치), 순서는 항상 같다:

1. **해부**: 그 업무를 단계로 쪼개고 "자동화할 것 / 내가 직접 할 것"을 구분한다(강의 1단원의 도구). 10분짜리 메모면 충분.
2. **절차화**: 반복되는 부분을 SKILL.md로 쓴다 — 트리거, 입력, 출력, 판단 기준, 실패 시 행동. `skills/QUALITY-CHECKLIST.md`(P3-S1 산출물)를 통과시킨다.
3. **역할이 필요하면**: `roles/`에 책임·금지사항·산출물을 적는다. 기존 역할 명세를 복제해 고치는 게 가장 빠르다.
4. **품질이 누적돼야 하면**: Loop로 감싼다(초안→리뷰→수정→재검토 패턴 재사용).
5. **버튼이 필요하면**: 대시보드 화이트리스트에 액션 추가 + ENGINE.md 매핑표 갱신 + 디스패처 지시문 반영. (이 3곳이 항상 같이 움직여야 한다 — 하나만 고치면 F-01형 드리프트가 재발한다.)
6. **월간 점검**: `ops/OPERATIONS.md`(P6-S2)의 V-시리즈 재실행이 건강검진이다. 문서와 실물의 어긋남(드리프트)이 이 시스템의 유일한 만성병이고, 점검이 유일한 약이다.

## 4. 앞으로 모델과 일할 때 — 변경 관리 5스탭

Fable 세션이 없어도 괜찮도록 설계는 "모델이 덜 똑똑해도 시스템이 똑똑하게"를 지향했다. 문서가 기억을, 게이트가 판단을, 원장이 연속성을 대신한다. 설계를 발전·보완·수정할 때는 이 절차를 지켜라:

- **Step 1 — 변경도 접수부터.** 어떤 모델에게든 "바로 고쳐"라고 하지 말 것. 먼저 새 브랜치에서 [왜 / 무엇을 / 영향 범위] 3줄짜리 변경 제안을 쓰게 하라.
- **Step 2 — 정합성 질문을 명시적으로.** "이 변경이 05의 발견 F-xx, 02의 단계 P-xx, ENGINE.md 매핑과 충돌하는가?"를 반드시 시켜라. 약한 모델은 시키지 않으면 교차 확인을 생략한다.
- **Step 3 — 4요소 없으면 실행 금지.** 백업/수정/검증/롤백이 없는 변경안은 반려하라. 형식 자체가 안전장치다(06의 모든 FIX가 이 형식인 이유).
- **Step 4 — 교차 검증.** 작성한 세션이 아닌 **다른 세션**(가능하면 다른 모델)이 검수한다. 04 킥오프를 재사용하면 된다.
- **Step 5 — 문서 동기화로 마무리.** 코드·스킬만 고치고 00 지도·ENGINE.md·역할 명세를 안 고치면 그 변경은 다음 세션에게 보이지 않는 부채가 된다. "문서 갱신까지가 변경"이다.

## 5. 모델별 실패 패턴과 방어 (실전 수칙)

| 실패 패턴 | 증상 | 방어 습관 (킥오프에 이미 내장된 것 포함) |
|---|---|---|
| 과잉 확신 | 검증 없이 "완료했습니다" | "됐다"가 아니라 **출력을 붙여라**를 요구. 07의 증거 열이 그 강제장치 |
| 컨텍스트 망각 | 긴 세션 후반에 초기 규칙 위반 | 세션을 짧게 끊고 원장으로 재개. 중요한 규칙은 "03을 다시 읽어라" 한 줄로 리로드 |
| 범위 이탈 | 시키지 않은 "개선"을 곁들임 | "02에 없는 작업은 실행 말고 제안만"(03에 내장). 리뷰 때 diff에서 요청 밖 변경을 찾아라 |
| 중복 구축 | 이미 있는 걸 다시 만듦 | 착수 전 "기존 대응물을 먼저 검색해 보고하라" 요구(R14 패턴) |
| 삭제·덮어쓰기 사고 | 백업 없이 원본 수정 | 삭제 금지·백업 규칙을 킥오프마다 반복(이미 03·전역 CLAUDE.md에 있음). .bak 확인 후 진행 |
| 동조(아첨) | 당신의 잘못된 제안도 그대로 수행 | 설계 논의 때 "내 제안의 약점 3개부터 말하라"를 먼저 시켜라 |
| 환각 참조 | 존재하지 않는 파일·스킬 언급 | "그 파일 경로를 ls로 증명하라" — 존재 증명 요구를 습관화 |

## 6. 모델 사용 전략과 시스템의 헌법

**모델 배치**: 일상 실행(배치·정리·회신)은 Sonnet급이면 충분하다 — 절차가 SKILL.md로 문서화될수록 요구 지능이 내려간다(그게 스킬의 존재 이유다). 설계 변경·감사·게이트 판정은 그 시점의 최상위 모델에게. 연 1~2회의 큰 개편은 이번 세션의 방법 그대로: **탐색(실측) → 감사(05 형식) → 계획(02 형식) → 위임(03/04 형식)** — 이 4단 방법론 자체가 repo에 남아 있으므로 어떤 모델이든 재연할 수 있다.

**이 시스템의 헌법 4조 (모델이 바뀌어도 불변)**:
1. 우선순위: 안전 규칙(삭제 금지·백업·2회 실패 중단) > 위임 문서의 자율권 > 전역 "질문 후 진행".
2. 단일 기록자: 07은 빌더만, 08은 검수자만. 상태는 대화가 아니라 파일에.
3. 게이트: 검수 PASS 없이 다음 구간 없음. 실기동 증거 없이 G2 없음.
4. 증거 기반 완료: 관찰 가능한 출력이 없으면 완료가 아니다.

**시스템 특유의 함정(기술 각론)**: Gist는 정기 회차 시간대(±20분)에 만지지 말 것 · 타임스탬프는 +09:00 저장, 비교는 epoch, 문자열 비교 금지 · 공개 repo이므로 ID·개인정보는 "gist-config.md 참조"로 간접 표기 · claude.ai 스킬 반영은 웹에서만 가능([USER] 단계) · 경로는 devices/ 프로파일 경유 · 문서-실물 드리프트가 최대 적, 월간 점검이 방어선.
