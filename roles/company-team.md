# 회사 팀 5역할 — 책임·금지사항·충돌 방지

> 출처: `team-room.html`(dev/research/business 3개 회의실), `index.html`(secretary 액션 `company.brief`). 여기서는 실행 코드를 바꾸지 않고 각 역할의 계약만 명문화한다.

## 1. 개발팀 (dev)
- **책임**: eBookshelf/Hubrary·VoiceReading 프로젝트 레지스트리 최신화. `dev.verify`(run-verify 검증), `dev.implement`(지금/예약 구현) 요청을 받아 실행하고 worklog에 결과를 남긴다.
- **금지사항**: 검증(`dev.verify`) 없이 구현 결과를 "완료"로 표시하지 않는다. 프로덕션 배포·외부 서비스 설정 변경은 이 역할의 자동 실행 범위 밖(사용자 실행).
- **충돌 방지**: 같은 프로젝트에 대해 연구팀·사업팀이 동시에 리서치를 요청해도, 코드 구현 자체는 개발팀 단독 소관 — 다른 팀은 요구사항만 전달한다.

## 2. 연구팀 (research)
- **책임**: 학술 파이프라인(inbox·master_index·위키) 운영. `research.batch`(배치 실행), `research.reports`(보고서 배치), `research.index`(인덱스 점검), `research.wiki`(위키 노트 소급)을 실행하고 LLM Wiki(`docs/plan` P2)에 결과를 남긴다.
- **금지사항**: 원문을 확보하지 못한 문헌에 대해 요약·인용을 단정하지 않는다(Literature Reviewer 서브에이전트와 동일 금지사항 공유 — P3-S4). Wiki index 갱신 없이 "배치 완료"로 표시하지 않는다.
- **충돌 방지**: 사역팀의 `ministry.research` 요청은 연구팀의 Literature Reviewer 부품을 재사용하되, 결과 회신처는 교회 Gist(사역 보드)로 한정 — 회사 위키에는 별도 사본을 남기지 않는다(중복 방지).

## 3. 사업팀 (business)
- **책임**: Hubrary 출시 준비, 사업 리서치. `biz.launch`(출시 체크 갱신), `biz.research`(주제 입력 리서치)를 실행.
- **금지사항**: 출시(`biz.launch`)의 "체크 갱신"은 상태 점검이지 실제 배포·과금·계약 체결이 아니다 — 그런 행위는 항상 사용자 실행.
- **충돌 방지**: 사업 리서치가 연구팀의 학술 파이프라인과 주제가 겹치면(예: 동일 저자·기술), 학술 산출물(Wiki)을 참조만 하고 사본을 만들지 않는다.

## 4. 비서 (secretary / company.brief)
- **책임**: 회사 전체 브리핑 생성(`company.brief`) — 팀별 상태·적체·실패 요약을 하나의 보고서로 종합.
- **금지사항**: 브리핑에 개인정보(실명·연락처)를 원문 그대로 인용하지 않는다. 외부 발송(이메일 등)은 브리핑 "생성"과 분리된 별도 승인 단계(F-14, P4-S5(b) 결정 대기).
- **충돌 방지**: 각 팀 Gist/데이터를 읽기만 하고 쓰지 않는다 — 비서 역할이 팀 상태를 임의로 변경하지 않는다.

## 5. 디스패처 총괄 (상위 조율)
- **책임**: 1분 폴링 + 정기 회차(04/18/23 KST) 스케줄링, 4개 팀 역할로의 요청 라우팅, 하트비트 기록(FIX-11 이후).
- **금지사항**: 화이트리스트 밖 액션 자동 실행 금지(CLAUDE.md §4 안전 보류 기준 그대로 적용).
- **충돌 방지**: 여러 팀의 요청이 동시에 도착하면 접수 순서(FIFO)를 따르되, 정기 회차 시간대(±20분)는 Gist 쓰기 충돌 회피를 위해 신규 수동 개입을 지양한다(R6).
